### 无障碍（A11y）与 WCAG 2.1 AA 深度探索

因一个客户项目要求必须通过无障碍合规审查，需要符合 WCAG 2.1 AA 标准，于是对该领域进行了深入研究。WCAG（Web Content Accessibility Guidelines）是目前最权威的 Web 无障碍标准，分为 A、AA、AAA 三个等级。大多数企业的合规线定在 AA 级。

WCAG 的核心是 POUR 四大原则：

**可感知（Perceivable）**：用户必须能"看到/听到"内容。关键实现包括：所有 `<img>` 必须有 `alt` 属性；视频必须提供字幕；不能仅靠颜色传达信息（如红色=错误）。对比度要求是 AA 的核心硬指标——普通文本 ≥ 4.5:1，大文本 ≥ 3:1。

**可操作（Operable）**：用户必须能"操作"。重点在于键盘可访问性——所有交互功能必须支持 Tab/Enter/Space 操作，必须有明确的 `:focus` 样式。Modal 组件必须实现 focus trap（Tab 不能逃逸）且 ESC 可关闭。页面缩放 200% 不影响使用，在 320px 宽度下无横向滚动。

**可理解（Understandable）**：用户必须"看得懂"。表单每个 input 必须有 label，错误提示需通过 `aria-describedby` 关联到具体输入框，且提供修复建议。UI 行为需一致且符合预期。

**健壮性（Robust）**：要兼容屏幕阅读器等辅助工具。使用语义化 HTML，正确使用 ARIA 属性（但不过度使用），动态内容更新使用 `aria-live`。

在工具链方面，评估了几个关键方案：`eslint-plugin-jsx-a11y` 用于静态代码分析，在开发阶段捕获 JSX 中的无障碍问题；`@axe-core/react` 用于运行时检测，结果直接输出到 Chrome DevTools 控制台；`jest-axe` 可集成到 CI 流水线做自动化检测。在自定义 React 组件方面，优先考虑 React Aria 和 Radix UI 等已经内置无障碍支持的库。

### Deep Dive into Accessibility (A11y) and WCAG 2.1 AA

Due to compliance requirements on a client project, a thorough exploration of web accessibility standards was conducted. The project required full compliance with WCAG 2.1 AA — the standard most enterprises use as their compliance baseline, sitting between the basic (A) and the most strict (AAA) levels.

WCAG is built on four core principles known as POUR:

**Perceivable**: Users must be able to "see/hear" content. Key implementations include: every `<img>` must have an `alt` attribute; videos must include captions; information must not be conveyed by color alone (e.g., red ≠ error). The critical AA metric is contrast ratio — normal text requires ≥ 4.5:1, large text ≥ 3:1.

**Operable**: Users must be able to "interact." The primary focus is keyboard accessibility — all interactive elements must be reachable and activatable via Tab/Enter/Space, with visible `:focus` styles. Modal components must implement focus trap (Tab cannot escape the modal) and ESC-to-close. Pages must remain usable at 200% zoom and produce no horizontal scroll at 320px viewport width.

**Understandable**: Users must be able to "comprehend." Every form input requires a label, error messages must be linked via `aria-describedby`, and repair suggestions should be provided. UI behavior must be consistent and predictable.

**Robust**: Content must work with assistive technologies like screen readers. This means semantic HTML, correct (but not excessive) ARIA usage, and `aria-live` regions for dynamic content updates.

Several toolchain options were evaluated: `eslint-plugin-jsx-a11y` for static analysis of JSX accessibility issues during development; `@axe-core/react` for runtime detection with results in Chrome DevTools; and `jest-axe` for CI pipeline integration. For custom React components, libraries like React Aria and Radix UI were identified as preferred options due to their built-in accessibility support.
